MiniDone Security Review

Last reviewed: June 14, 2026

Status: no confirmed data exfiltration, secret leakage, network, WebView, shell execution, or credential-handling findings were found in the reviewed codebase.

MiniDone is local-first. Release packaging now produces a clean app zip and rejects debug/test artifacts before upload.

Stored Data

  • Tasks: title, status, dates, ordering, recurrence, project, and tags.
  • Projects: name, creation date, and color index.
  • Tags: name and creation date.
  • Preferences: language, theme, mini-window height, and onboarding completion state.

Checks

  • No account login, cloud sync, analytics, WebView, backend calls, or hardcoded secrets.
  • No shell/process execution APIs, keychain usage, or credential handling.
  • UI-test hooks are compiled only in Debug builds.
  • Drag-and-drop no longer publishes task titles as plain text.
  • Release builds use a minimal App Sandbox entitlement.
  • Release zip rejects .xctest, .xcresult, .dSYM, DerivedData, test targets, __MACOSX, and ._ metadata files.
  • Release packaging rejects the debug get-task-allow entitlement and UI-test environment hooks in the compiled binary.

Residual Risks

  • Local SwiftData files are not app-level encrypted. Malware or a user with filesystem access to the account may be able to read local data.
  • GitHub always exposes automatic source archives for tags; normal users should download the attached MiniDone-macOS-vX.Y.Z.zip release asset.
  • Public distribution still needs Developer ID signing and notarization for the smoothest Gatekeeper experience.

Release Checklist

  • Run unit tests.
  • Run UI tests.
  • Run scripts/package_release.sh.
  • Upload the generated dist/MiniDone-macOS-vX.Y.Z.zip asset to GitHub Releases.
  • Use Developer ID signing and notarization before broad public distribution.
